### 1.1 三大模型定位

#### 白話解釋

想像你要找人幫忙做事：

| 模型 | 類比 | 白話 |

| **Haiku 4.5** | 實習生 | 最快最便宜，簡單任務秒完成 |

> TIP：**不確定選哪個？先用 Sonnet。** Sonnet 能處理 80% 的日常任務，只有在結果明顯不夠好時才需要升級到 Opus。

### 1.2 什麼是 Context Window？

#### 白話解釋

Context Window（上下文窗口）= **Claude 一次能「看」多少東西**。

把 Claude 想像成一個人坐在桌子前。Context Window 就是桌子的大小：
- **桌子越大**，能同時攤開的文件越多
- **桌子越小**，需要不斷收起舊文件才能看新的

| 模型 | Context Window | 白話容量 |
|------|---------------|---------|
| Opus / Sonnet | 1M tokens | ~2,500 頁 A4 文件 |
| Haiku | 200K tokens | ~500 頁 A4 文件 |

#### 1M tokens 有多大？                                           COMPARE

```
1M tokens 大約等於：
├── 2,500 頁 A4 文件
├── 一整個中型專案的程式碼（~50,000 行）
├── 5 本 200 頁的書
├── 一年份的所有電子郵件
└── 一部小說的完整手稿

對比其他 AI：
├── GPT-4：128K（Claude 的 1/8）
├── Gemini 1.5：2M（比 Claude 大 2 倍）
├── Claude：1M（中間值，但品質最穩定）
└── 大部分開源模型：8K-128K
```

#### token 是什麼？

- **英文：** 1 個 token ≈ 0.75 個英文單字（1 word ≈ 1.3 tokens）
- **中文：** 1 個中文字 ≈ 1.5-2 個 tokens（中文比英文貴 2-3 倍！）
- **程式碼：** 1 行程式碼 ≈ 10-15 個 tokens

> WARNING：中文消耗的 token 是英文的 2-3 倍。同樣的問題，用中文問比用英文問貴 2 倍。如果你追求省錢，可以用英文下指令、要求中文輸出（→ Ch26.6）。

→ Ch25 有完整的 token 計算公式和各語言的精確換算表。

### 1.3 Adaptive Thinking（自適應思考）

#### 白話解釋

以前的 Claude 有一個「延伸思考」開關——開了就會多想一下。

現在的 Claude 4.6 更聰明：**它會自己判斷問題有多難，自動決定要想多久。** 這就是 Adaptive Thinking（自適應思考）。

| 問題難度 | Claude 的反應 | 類比 |
|---------|-------------|------|
| 「你好」 | 幾乎不思考，直接回答 | 反射動作 |
| 「幫我翻譯這段話」 | 簡單思考 | 正常對話 |
| 「設計一個分散式快取系統」 | 深度思考，花較多時間 | 認真研究 |
| 「證明這個數學定理」 | 最深度思考，花最多時間和 token | 全力以赴 |

#### Effort 等級（API / CLI 使用者）

如果你想手動控制 Claude 的思考深度：

| Effort | 白話 | 思考深度 | Token 消耗 | 適合場景 |
|--------|------|---------|-----------|---------|
| `low` | 快問快答 | 最淺 | 最少 | 翻譯、格式轉換 |
| `medium` | 正常思考 | 中等 | 中等 | 日常任務（預設） |
| `high` | 仔細推敲 | 深 | 較多 | 程式碼審查、分析報告 |
| `max` | 全力思考 | 最深 | 最多 | 數學證明、複雜架構 |

```bash                                                          [可複製]
# 在 Claude Code 對話中切換 effort
/effort low       # 切到快問快答模式
/effort high      # 切到深度思考模式
```

> TIP：大部分情況下讓 Claude 自動判斷（Adaptive）就好。只有在你明確知道「這個問題很簡單，不需要想太久」或「這個問題非常重要，請全力思考」時，才手動調整 effort。

### 1.4 模型選擇決策樹

```
你要做什麼？
│
├─ 🟢 簡單任務（翻譯/摘要/分類/格式轉換）
│   └── → Haiku 4.5 ★☆☆
│       最快、最便宜。能做就不要用更大的模型。
│
├─ 🟡 一般任務（寫程式/寫文章/分析資料/問答）
│   └── → Sonnet 4.6 ★★☆
│       80% 的任務用 Sonnet 就夠。性價比最高。
│
├─ 🔴 困難任務（系統架構/複雜推理/重要決策/數學）
│   └── → Opus 4.6 ★★★
│       最強但最貴。只在 Sonnet 做不好時升級。
│
└─ ⚡ 大量重複任務（1,000+ 封郵件/批次處理）
    └── → Haiku 4.5 + Batch API
        半價 + 最快。不需要即時回應的批次任務首選。
```

> TIP：在 Claude Code 中，預設使用 Sonnet。你可以隨時用 `/model` 切換：
```bash                                                          [可複製]
/model opus       # 切到 Opus（複雜任務）
/model sonnet     # 切回 Sonnet（日常任務）
/model haiku      # 切到 Haiku（簡單任務）
```

## ⑥ 常見陷阱

> WARNING：新手最常犯的 3 個錯誤

### 陷阱一：一律用最強的 Opus

**問題：** 所有任務都用 Opus，月底帳單嚇死人。
**解法：** 80% 的任務用 Sonnet 就夠好。先用 Sonnet，結果不滿意再用 Opus 重跑——這樣比一律用 Opus 省 40-60%（→ Ch26）。

### 陷阱二：忽略中文的 token 消耗

**問題：** 以為中文和英文一樣便宜。其實中文消耗的 token 是英文的 2-3 倍。
**解法：** 重要提示詞用英文寫（省錢），要求 Claude 用中文回覆。或者接受多花的費用——品質更重要（→ Ch25）。

### 陷阱三：不知道有 Batch API

**問題：** 100 個相同任務一個一個手動送。
**解法：** 用 Batch API 批次送出，自動半價。不需要即時回應的任務（如資料分析、文件翻譯），永遠用 Batch（→ Ch26.5）。
